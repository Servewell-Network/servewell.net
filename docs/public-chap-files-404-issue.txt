Here's what I think is actually happening: the 19:29 deploy uploaded the chapter files as new assets (their hashes weren't in the previously-active manifest, so Wrangler uploaded them fresh). Cloudflare's asset delivery system serves newly-uploaded assets directly for up to 1 hour — but after that TTL, the CDN falls back to checking whether those asset hashes are still referenced by the current active worker binding. If the binding's manifest wasn't cleanly updated (perhaps because "already uploaded" from an earlier version was cached in the binding layer), those hash references expire and 404s resume.

This is conjecture about Cloudflare internals, but the pattern points to the asset manifest embedded in the worker itself being stale — not the CDN cache. The practical fix is the same either way: make sure the chapter files get newly regenerated (not hash-identical to old uploads) every time you deploy, so Wrangler is forced to re-upload them and update the manifest binding freshly. Adding a build timestamp comment to the phasing output would achieve that, but it's a sledgehammer. Alternatively, always running wrangler deploy after a fresh phasing run (which you already do) should be sufficient — the issue kept recurring because the .assetsignore was silently zeroing out.

The real danger was that .assetsignore kept becoming 0 bytes. With .DS_Store as real content and the pre-deploy guard checking the exact content, that specific trigger should be gone. I'd keep the monitor running for the next few deploys to confirm the 1-hour pattern doesn't recur.
